# 🎤 ملف المناقشة — Frontend (Avicena)

دليل للدفاع عن شغل الفرونت: نبذة سريعة، القرارات التقنية وليه، المشاكل وحلولها، وأسئلة متوقّعة بإجاباتها.

---

## 1) نبذة سريعة (30 ثانية)

> "بنيت الطبقة الأمامية (Frontend) لمنصة Avicena الطبية بـ **Next.js + TypeScript**، فيها نظام **مصادقة كامل** (تسجيل بخطوات + OTP + استعادة كلمة مرور) بـ **react-hook-form و zod**، مع **توجيه كل مستخدم لداشبورده حسب دوره** وحماية الصفحات، ونظام أمان يخزّن التوكن في **httpOnly cookies** بدل الـ localStorage. بالإضافة لصفحات عامة (رئيسية، أطباء، معامل) بتصميم مودرن responsive."

---

## 2) القرارات التقنية — وليه اخترتها (الأهم في المناقشة)

| القرار | ليه؟ |
|--------|------|
| **موقع واحد بأدوار (Route Groups)** بدل موقع لكل دور | الباك اند واحد، والأدوار بتتفاعل (شات/إشعارات). موقع واحد = كود أقل، صيانة أسهل، deploy واحد. |
| **react-hook-form + zod** للفاليديشن | فصل قواعد التحقق في **schema واحد** (zod) قابل لإعادة الاستخدام، و RHF يقلّل الـ re-renders ويدير الأخطاء تلقائياً. أنظف من الـ if/else اليدوي. |
| **httpOnly cookies** للتوكن (مش localStorage) | localStorage عرضة لـ **XSS** (أي سكربت يقراه). httpOnly cookie الجافاسكريبت ما يقدرش يقراه → أأمن، خصوصاً إننا بنتعامل مع **بيانات طبية**. |
| **Axios instance واحد + interceptor** | مكان واحد لكل الإعدادات (`withCredentials`, معالجة 401, refresh تلقائي) → باقي الكود ما يشغلش باله. |
| **Zustand** لإدارة الجلسة | أخف وأبسط من Redux لحالة صغيرة (session). |
| **Wizard متعدد الخطوات للتسجيل** | نقسّم فورم طويل لخطوات مفهومة (حساب → بيانات → تحقق) → تجربة أفضل ومعدّل إكمال أعلى. |

## 3) المشاكل اللي واجهتها وحليتها (يبيّن مهارة)

1. **شكل الـ API مختلف عن المتوقّع** — الباك بيرجّع البيانات في المستوى الأعلى مش جوه `data`. → ظبطت الفرونت على الشكل الحقيقي.
2. **CORS** — الباك مكانش بيسمح للفرونت. → اتحل + تعلّمت إن السيرفرات محتاجة restart لتحميل التغييرات.
3. **تكامل مع شغل الفريق** — لما الفريق حوّل الباك لـ **JWT + httpOnly cookies**، عدّلت الفرونت بالكامل: `withCredentials`، شيل التوكن من localStorage، وrefresh تلقائي عند 401.
4. **إدارة الـ Git** — اشتغلت على فرع منفصل، ودمجت شغل الفريق مع شغلي وحليت التعارضات من غير ما أخسر حاجة.

## 4) الأسئلة المتوقّعة + الإجابات

**س: ليه react-hook-form مش فاليديشن يدوي؟**
ج: RHF بيدير الفورم بأداء أعلى (re-renders أقل)، و zod بيخلّي القواعد **schema واحد** يتغيّر من مكان واحد. الفاليديشن **live** (`mode: onTouched`) — الأخطاء تظهر وانت بتكتب.

**س: ليه cookies مش localStorage للتوكن؟**
ج: أمان. localStorage معرّض لـ XSS. httpOnly cookie الجافاسكريبت مايقراهوش، والمتصفح يبعته تلقائياً مع `withCredentials`. مهم في تطبيق طبي.

**س: إزاي بتوجّه كل مستخدم لداشبورده؟**
ج: الباك يرجّع `role` عند الدخول → أخزّنه → `homeFor(role)` يرجّع صفحة الدور → `router.replace`. وأحمي الصفحات بـ **RoleGuard** (الداخل الغلط يترفض) و **GuestGuard** (الداخل مايشوفش صفحات الدخول).

**س: لو التوكن خلص وانت بتعمل طلب؟**
ج: الطلب يرجّع 401 → الـ interceptor يستدعي `/api/auth/refresh` (يجدّد الكوكي) ويعيد الطلب تلقائياً — من غير ما المستخدم يحس.

**س: إزاي بتتعاملي مع الأخطاء؟**
ج: دالة موحّدة `getErrorMessage` بترجّع رسالة عربية، وبتعالج **429** (محاولات كثيرة) بشكل خاص.

**س: الموقع responsive؟**
ج: أيوه، عربي/RTL افتراضياً، وكل صفحة بتتأقلم بالـ breakpoints (موبايل → ديسكتوب) — مش نسخ منفصلة.

## 5) إزاي أعمل Demo (خطوات العرض)

1. **الرئيسية** → أعرض التصميم المودرن (responsive: كبّر/صغّر).
2. **إنشاء حساب** → أبيّن الـ Wizard (3 خطوات + stepper) والفاليديشن الـ live.
3. **تأكيد الإيميل** → OTP + العدّاد.
4. **تسجيل الدخول** → أدخل بحساب → يوجّهني للداشبورد حسب الدور.
5. **الحماية** → أجرّب أفتح صفحة دور تاني → يرفض (RoleGuard).
6. **DevTools → Application → Cookies** → أبيّن إن التوكن في **httpOnly cookie** مش localStorage.

## 6) اللي لسه (بصراحة — يبيّن وعي)

- **الداشبوردات الداخلية** لسه UI فاضي — الخطوة الجاية.
- **باگ في الباك** (أسماء الكوكيز في الـ guards) لازم الفريق يصلّحه عشان الطلبات المحمية تشتغل.
- **CSRF** مش متضاف لسه (مع الكوكيز يُفضّل إضافته لاحقاً).
