# Avicena — شرح ما تم إنجازه (للمناقشة)

مستند يشرح خطوة بخطوة اللي اتعمل في المشروع، ليه اتعمل كده، والقرارات التقنية المهمة. مرتّب كنقاط للعرض والنقاش.

---

## 1. نقطة البداية: فهم المشروع

**Avicena** منصة رعاية صحية عن بُعد (Telemedicine) فيها **4 أنواع مستخدمين**: مريض، دكتور، أدمن، معمل.

- الـ **Backend** جاهز ومبني بشكل احترافي (Node.js + Express، معماري modular).
- بدأنا بقراءة **كل الـ API** من الـ Postman collection + الكود الفعلي (الموديلات، الراوتس، السيرفسز) عشان أي شغل بعد كده يكون مبني على الحقيقة مش تخمين.

**التقنيات في الباك اند:** Express 5 · MongoDB (Mongoose) · Redis · BullMQ (queues) · Socket.io (real-time) · Qdrant (vector DB للـ AI) · Cerebras + Gemini (المساعد الطبي RAG) · Stripe/Paymob · Cloudinary.

---

## 2. القرار المعماري الأول: موقع واحد أم مواقع منفصلة؟

**القرار:** موقع واحد (Single App) بدومين واحد، والداشبورد بتتغيّر حسب دور المستخدم — **مش** موقع منفصل لكل نوع.

**الأسباب:**
- الباك اند واحد لكل الأدوار.
- فيه تفاعل بين الأدوار (شات، فيديو، إشعارات) — لو فصلنا هنكرّر الكود.
- صيانة أسهل، design system واحد، deploy واحد، تكلفة أقل.

**التنفيذ:** فصل الأدوار عن طريق **Route Groups** جوه نفس التطبيق:
`/patient/*` · `/doctor/*` · `/admin/*` · `/lab/*` — كل مجموعة محميّة بحارس (Guard) يتأكد من الدور.

## 4. هيكل الفرونت اند (Next.js)

عملنا scaffold احترافي بـ **Next.js (App Router)** منظّم:

```
src/
├── app/                  # الصفحات (route groups لكل دور)
├── features/<domain>/    # الربط بالـ API لكل موديول
├── lib/api/client.ts     # Axios + interceptor  ← أهم ملف
├── lib/auth/ · socket/   # الجلسة + real-time
├── store/auth.store.ts   # إدارة المصادقة
└── config/roles.ts       # خريطة الأدوار
```

**قرار مهم:** Arabic-first / **RTL** بشكل افتراضي (لأن المحتوى الطبي عربي).

## 5. سير عمل التصميم عبر Google Stitch

بدل ما نصمّم من الصفر، استخدمنا **Google Stitch** (أداة تصميم بالـ AI):

- عملنا **جرد كامل بـ 51 شاشة** مربوطة كل واحدة بالـ endpoint بتاعها (`STITCH-PROMPTS.md`) — عشان ولا فيتشر في الباك اند تضيع.
- قسّمناها لـ **9 batches** (مجلد `stitch/`)، كل batch = جلسة تصميم واحدة عشان الشكل يفضل متناسق.
- **الاكتشاف المهم:** الـ 51 شاشة فعلياً = **~10 قوالب متكررة** بس (جداول، فورمات، داشبوردات) — فمش هنصمّم 51 تصميم منفصل.

## 6. تنفيذ المصادقة (Login) — وأهم تحدّي اتحلّ

### التحدّي: الباك اند فيه 4 أنظمة توكن مختلفة
كل دور بيبعت التوكن في **header مختلف**:
- المريض → `Authorization: Bearer`
- الدكتور → `dtoken` · الأدمن → `atoken` · المعمل → `ltoken`

**الحل في الفرونت:** جمّعنا التعقيد ده في مكان واحد:
- `config/roles.ts` — خريطة: كل دور → الـ header بتاعه.
- `lib/api/client.ts` — الـ interceptor بيحط الـ header الصح **تلقائياً** حسب الدور المخزّن، فباقي الكود مش بيشغل باله.

### التحدّي الثاني: التصميم طلب "لوجين موحّد" (من غير اختيار نوع)
تصميم Stitch كان لوجين واحد من غير ما المستخدم يختار نوعه — بس الباك اند كان عنده 4 endpoints منفصلة للّوجين.

**الحل:** أضفنا **endpoint موحّد** في الباك اند (`POST /api/auth/login`) بيكتشف نوع المستخدم تلقائياً:
1. يدوّر في جدول المستخدمين (مريض/أدمن)
2. لو ملقاش، يدوّر في الأطباء
3. لو ملقاش، يدوّر في المعامل
4. يرجّع `{ token, role, user }` — والفرونت يخزّن الدور ويوديه للداشبورد الصح.

**ده بالظبط اللي التصميم محتاجه، وهو كمان الطريقة الأنضف هندسياً.**

## 7. المشاكل اللي واجهتنا وحليناها

| المشكلة | السبب | الحل |
|---------|-------|------|
| **شكل الـ response مختلف** | الباك اند بيعمل spread للبيانات في المستوى الأعلى (`token` مش `data.accessToken`) وكل دور بـ object باسم مختلف | ظبطنا الفرونت على الشكل الحقيقي |
| **CORS blocked** | الباك اند مكانش فيه `cors` middleware أصلاً | أضفناه + سطّبنا الحزمة |
| **زرار/ألوان مش ظاهرة** | الـ dev server مابيلتقطش تغييرات كونفيج Tailwind أوتوماتيك | لازم restart للسيرفر |
| **مفيش refresh token عند اللوجين** | الـ service عنده الوظيفة بس مش متوصّلة | التوكن صالح 7 أيام (كافي مؤقتاً) — نفعّله لاحقاً |

**درس مهم:** الـ dev servers محتاجة **restart كامل** لما نغيّر dependencies جديدة أو ملفات كونفيج — مش hot-reload.
